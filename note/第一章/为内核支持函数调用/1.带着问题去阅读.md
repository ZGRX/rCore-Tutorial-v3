这节的核心问题其实是：**CPU 已经能执行 `_start` 了，但 Rust 函数调用依赖栈、返回地址、寄存器保存等机制；所以内核必须先手动建立一小段“函数调用环境”，再跳进 `rust_main`。** 这一节最后会在 `entry.asm` 里设置 `sp = boot_stack_top`，再 `call rust_main`，并在 Rust 入口里清零 `.bss` 段。

你可以带着这几个问题读：

1. **为什么内核不能一开始就直接写 Rust？**  
    重点看：在进入 Rust 之前，哪些运行条件还不存在？尤其是栈为什么必须先准备好。
    
2. **函数调用和普通跳转的本质区别是什么？**  
    普通跳转只要改 `pc`，但函数调用还必须知道“返回到哪里”。RISC-V 用 `jal/jalr` 保存返回地址到 `ra`，`ret` 再跳回 `ra` 指向的位置。
    
3. **为什么 `ra` 会在嵌套函数调用中出问题？**  
    思考这个场景：`f()` 调 `g()`，而 `g()` 又调 `h()`。如果 `ra` 被覆盖，`g()` 返回时还能不能回到 `f()`？这就是为什么要保存函数调用上下文。
    
4. **“调用者保存”和“被调用者保存”到底解决了什么问题？**  
    重点理解：调用者和被调用者如何分工保存寄存器；哪些寄存器由谁负责；保存和恢复发生在函数的开场白和结尾。
    
5. **RISC-V 的寄存器约定为什么这样分？**  
    重点记住：`a0~a7` 传参数和返回值，`t0~t6` 是调用者保存，`s0~s11` 是被调用者保存；`ra`、`sp`、`fp` 和函数调用关系最密切。
    
6. **栈帧是什么？为什么函数调用天然适合用栈？**  
    注意栈的“后进先出”：最近调用的函数最先返回，所以每次函数进入时分配栈帧，返回时释放栈帧。RISC-V 中栈从高地址向低地址增长。
    
7. **`entry.asm` 里这两行到底完成了什么？**
    
    ```asm
    la sp, boot_stack_top
    call rust_main
    ```
    
    第一行把 `sp` 设置到启动栈顶，第二行把控制权交给 Rust 的 `rust_main`。读的时候要问自己：如果没有第一行，Rust 函数调用会发生什么？
    
8. **为什么启动栈放在 `.bss.stack`，而且是 64KiB？**  
    重点理解它只是预留一块不会和代码、数据冲突的内存区域；栈不需要预先清零，因为函数调用会覆盖栈帧内容。(
    
9. **为什么 `rust_main` 要加 `#[no_mangle]`？**  
    因为汇编代码要按符号名找到 `rust_main`；如果 Rust 编译器改了符号名，链接时就找不到这个入口。
10. **为什么进入 Rust 后第一件事是清零 `.bss`？**  
    `.bss` 存放需要初始化为 0 的全局数据；在使用这些全局变量前，内核必须主动把 `sbss..ebss` 这段内存清零。
    

读的时候可以抓一条主线：**为了让 Rust 函数能正常调用和返回，内核先在汇编里准备栈；之后编译器才能自动生成函数调用所需的保存/恢复代码。**
