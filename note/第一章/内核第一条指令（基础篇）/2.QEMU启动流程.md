**QEMU 先从固定地址 `0x1000` 执行一小段内置启动代码，然后跳到 `0x80000000` 执行 RustSBI，RustSBI 初始化完成后再跳到 `0x80200000` 执行你的 rCore 内核。**

整体流程是：

```
QEMU 内置启动代码    PC = 0x1000        ↓跳转到 0x80000000        ↓RustSBI / bootloader        ↓跳转到 0x80200000        ↓rCore 内核 os.bin
```

## 1. QEMU virt 平台的物理内存布局

在 QEMU 模拟的 RISC-V `virt` 平台上，默认物理内存从：

```
0x80000000
```

开始。

默认大小是：

```
128 MiB
```

所以默认物理内存范围是：

```
[0x80000000, 0x88000000)
```

注意这是左闭右开区间：

```
包含 0x80000000不包含 0x88000000
```

也就是说，合法的默认物理内存地址大致是：

```
0x80000000 ~ 0x87ffffff
```

## 2. 启动前 QEMU 会先加载两个文件

启动之前，QEMU 会把两个二进制文件放到物理内存的指定位置：

```
0x80000000: rustsbi-qemu.bin0x80200000: os.bin
```

也就是：

```
物理内存0x80000000 ┌────────────────────┐           │ RustSBI bootloader │           │ rustsbi-qemu.bin   │           └────────────────────┘0x80200000 ┌────────────────────┐           │ rCore kernel       │           │ os.bin             │           └────────────────────┘0x88000000 ─────────────────────
```

`0x80200000 - 0x80000000 = 0x200000`，也就是 **2 MiB**。

所以 RustSBI 和内核之间隔了 2 MiB 空间。这样可以避免内核镜像覆盖 RustSBI，同时也符合 RustSBI 对下一阶段入口地址的约定。

## 3. 第一阶段：QEMU 内置启动代码

QEMU 启动后，CPU 的 `PC` 会被初始化为：

```
PC = 0x1000
```

也就是说，QEMU 执行的第一条指令在：

```
0x1000
```

这里不是你的内核，也不是 RustSBI，而是 **QEMU 内部固化的一小段启动代码**。

这段代码做的事情很少，核心目的就是跳到：

```
0x80000000
```

所以第一阶段可以理解为：

```
PC = 0x1000执行几条 QEMU 固化的指令跳转到 0x80000000
```

而 `0x80000000` 正好是 RustSBI 被加载的位置。

## 4. 第二阶段：RustSBI 启动

因为 QEMU 第一阶段固定跳转到：

```
0x80000000
```

所以必须把 RustSBI 放在这里。

于是 CPU 来到：

```
PC = 0x80000000
```

开始执行 RustSBI 的第一条指令。

RustSBI 的职责主要是：

```
初始化底层环境设置特权级准备进入 S-mode提供 SBI 调用接口最后跳转到内核入口
```

在 rCore 里，RustSBI 通常运行在：

```
M-mode
```

而 rCore 内核运行在：

```
S-mode
```

所以特权级关系是：

```
M-mode: RustSBIS-mode: rCore 内核U-mode: 用户程序
```

RustSBI 初始化完成之后，会跳转到它约定好的下一阶段入口地址：

```
0x80200000
```

所以第二阶段可以理解为：

```
PC = 0x80000000执行 RustSBI初始化机器环境切换/准备进入 S-mode跳转到 0x80200000
```

## 5. 第三阶段：rCore 内核启动

你的内核镜像 `os.bin` 被提前加载到了：

```
0x80200000
```

所以当 RustSBI 跳转到这个地址时，CPU 就开始执行 rCore 内核的第一条指令。

也就是：

```
PC = 0x80200000开始执行 os.bin
```

这说明控制权已经从 RustSBI 移交给了你的内核。

第三阶段就是：

```
PC = 0x80200000执行 rCore 内核第一条指令进入内核初始化流程
```

## 6. 三个关键地址

你只要记住这三个地址，整段流程就清楚了：

|地址|谁在这里|作用|
|---|---|---|
|`0x1000`|QEMU 内置启动代码|QEMU CPU 复位后的第一条指令地址|
|`0x80000000`|RustSBI|bootloader 入口地址|
|`0x80200000`|rCore 内核|内核入口地址|

对应流程：

```
0x1000  ↓0x80000000  ↓0x80200000
```

## 7. 为什么 RustSBI 放在 `0x80000000`？

因为 QEMU 第一阶段会固定跳转到：

```
0x80000000
```

所以如果你想让 RustSBI 作为 bootloader 运行，就必须让：

```
0x80000000 处保存 RustSBI 的第一条指令
```

否则 QEMU 跳过去之后就会执行错误内容。

换句话说：

```
QEMU 固定跳到 0x80000000所以 RustSBI 必须放在 0x80000000
```

## 8. 为什么内核放在 `0x80200000`？

因为 RustSBI 约定下一阶段软件，也就是内核，入口地址是：

```
0x80200000
```

所以你必须保证：

```
0x80200000 处保存内核的第一条指令
```

这通常还要求你的链接脚本也配合这个地址。

例如内核的链接地址应该从：

```
BASE_ADDRESS = 0x80200000
```

开始。

否则会出现一种典型错误：

```
QEMU 把 os.bin 加载到了 0x80200000但是 os.bin 内部认为自己运行在别的地址
```

那跳转、全局变量访问、函数地址计算都可能出问题。

所以这里有两件事要同时一致：

```
加载地址 = 0x80200000链接地址 = 0x80200000
```

## 9. 启动流程完整图

可以画成这样：

```
QEMU 启动  │  │  加载 rustsbi-qemu.bin 到 0x80000000  │  加载 os.bin 到 0x80200000  ↓CPU PC = 0x1000  │  │  QEMU 内置启动代码  ↓跳转到 0x80000000  │  │  RustSBI 开始执行  │  RustSBI 做初始化  │  RustSBI 准备进入 S-mode  ↓跳转到 0x80200000  │  │  rCore 内核开始执行  ↓进入内核初始化
```

## 10. 一句话版本

你可以这样背：

> QEMU 的 `virt` 平台默认物理内存从 `0x80000000` 开始。QEMU 复位后先从 `0x1000` 执行内置启动代码，然后固定跳到 `0x80000000`，所以 RustSBI 必须放在那里。RustSBI 初始化完成后按约定跳到 `0x80200000`，所以 rCore 内核 `os.bin` 必须被加载并链接到 `0x80200000`。
